# 超长篇治理协议

用于 100 万字以上，尤其是目标 300 万字级别的平台向网文连载。

启用判定补充：

- 如果项目目标总字数达到 100 万字，默认视为长篇治理范围
- 如果项目没有明确规划，但已写到约 30 万字或 60 章，也应自动转入治理闭环

这不是单章写作技巧，而是长篇工程协议。

## 1. 为什么必须额外治理

300 万字不是“把单章流程重复 1000 次”。

到这个规模时，最容易死的不是某一章写得差，而是：

- 主线漂移
- 角色弧线断裂
- 伏笔债务堆积
- 战力和法则污染
- 卷末没有结算
- 中段长期无回报
- 设定偷偷改了却没同步

所以默认要增加 4 类治理文件：

- `docs/全书宪法.md`
- `docs/卷纲.md`
- `docs/阶段规划.md`
- `docs/变更日志.md`

## 2. 长篇状态机

超长篇默认按下面的状态机运作，而不是直接“想到下一章就写”：

1. 全书立项
2. 建立全书宪法
3. 制定卷纲
4. 制定当前阶段规划
5. 用 `governance` 写入当前卷 / 当前阶段 / 阶段目标
6. `preflight`
7. `resume`
8. `plan`
9. `compose`
10. `start`
11. 写正文
12. `check`
13. `finish --summary`
14. 每 10-20 章一次 `audit --scope stage`
15. 每卷结束一次 `audit --scope volume`
16. 任何结构性改动先写 `变更日志`

跳步会直接提高漂移风险。

## 2.1 输入治理层

超长篇不要只靠“读了文档就开写”。

建议每章都先落 4 个运行时产物：

- `runtime/chapter-XXXX.intent.md`
- `runtime/chapter-XXXX.context.json`
- `runtime/chapter-XXXX.rule-stack.yaml`
- `runtime/chapter-XXXX.trace.json`

它们的作用：

- `intent`：告诉这一章真正要完成什么
- `context`：记录本章实际选入了哪些上下文
- `rule-stack`：记录本章优先遵守哪些规则
- `trace`：留下本章输入编排轨迹，便于复盘

## 3. 节拍要求

### 3.1 阶段节拍

默认每 10-20 章为一个阶段。

每个阶段都必须写清：

- 当前前台问题
- 阶段目标
- 最迟回报点
- 必须发生的关系变化
- 不能提前解决的问题
- 阶段结算后带走什么

### 3.2 卷节拍

每卷都必须有：

- 卷进入条件
- 卷核心任务
- 卷主反派 / 主阻碍
- 卷末兑现
- 卷末遗留问题
- 下一卷入口

### 3.3 变更节拍

凡是影响下面任一项的改动，都不能只在脑子里改：

- 全书终局
- 主角长期弧线
- 世界底层规则
- 长线关系边界
- 卷末结算
- 关键伏笔归属

必须先写进 `docs/变更日志.md`，再同步相关文档。

## 4. 审计标准

### 阶段审计

至少检查：

- 当前阶段是否还在推进原定问题
- 这一阶段是否出现长期无回报
- 当前关系线有没有真实变化
- 伏笔是在增债还是在回收
- 是否出现为推进剧情而强行降智

### 卷审计

至少检查：

- 本卷是否兑现了进入时承诺的看点
- 卷末是否既结算又留口
- 本卷新增设定有没有污染全书宪法
- 主角是否拿到了足够进入下一卷的新位置
- 读者是否能感到“这一卷写完了”，而不是平推结束

## 5. 主代理与辅助代理分工

### 主代理

负责：

- 最终方向拍板
- 全书宪法维护
- 卷末取舍
- 风格与审美终审

### 外部深度研究 / 审计辅助

如果你当前环境里有可用的深度研究代理、外部审计助手或并行分析工具，它们适合负责：

- 卷纲审计
- 阶段审计
- 时间线 / 伏笔债务 / 规则一致性
- 把散想法压成结构化中间件

### `copywriting`

如果当前环境可用 `copywriting` 这类表达优化能力，它更适合负责：

- 局部扩写 / 压缩 / 改口吻
- 章纲转场景卡
- 同场景多版本改写
- 局部表达优化

## 6. 什么时候必须停笔

出现任一情况，默认先停笔，不继续正文：

- `preflight` 失败
- 当前卷 / 当前阶段未记录
- 长线治理文件缺失
- 变更日志有待同步项
- 阶段审计失败但继续硬写
- 卷该结算却一直拖章

超长篇的核心不是“不断写”，而是“不断写而不崩”。
