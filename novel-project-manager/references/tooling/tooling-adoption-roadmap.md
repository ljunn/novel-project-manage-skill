# 外部工具吸收路线图

用于评估外部 AI 写作工具、开源项目或工作台时，哪些能力值得吸收进本仓库，哪些不值得直接照搬。

默认前提：

- 本仓库的核心不是“网页写作器”，而是平台向长篇网文连载工作流
- 吸收顺序优先级是：可复用流程资产 > 结构化提示词资产 > 脚本能力 > UI 外壳
- 不为了追功能数量，牺牲主线稳定性、一致性和可维护性

## 1. 优先吸收什么

### 1.1 提示词工作台

值得吸收：

- 最终提示词预览
- 提示词变量替换
- 提示词编辑后再执行
- 按任务类型切换提示词模板

原因：

- 它能把“提示词效果”从黑盒变成可调试对象
- 方便把稳定规则沉淀成可维护资产，而不是散落在对话里

在本仓库的落点：

- 通用规则写进 `SKILL.md`
- 可复用模板写进 `references/`
- 后续若做脚本或 UI，再从这些文本资产生成功能

### 1.2 定向返修微操作

值得吸收：

- 深化冲突
- 补人物动机
- 增加伏笔
- 压缩解释腔
- 强化结尾钩子
- 去 AI 味

原因：

- 比“整章重写”更稳
- 更符合长连载项目的低风险返修需求

在本仓库的落点：

- 统一写进 `references/quality/micro-revision-ops.md`
- 单章返修时优先选一个明确操作，不并行乱改

### 1.3 拆书导入能力

值得吸收：

- 导入 txt
- 自动识别章节
- 按章节拆分
- 把拆出的文本转成摘要、伏笔、时间线、人物与关系信息

原因：

- 它能把“外部成书学习”变成结构化资产
- 适合辅助逆向提炼，不适合直接拿来当正文生成器

在本仓库的落点：

- 未来可做独立脚本
- 输出优先对齐 `docs/项目总纲.md`、`docs/章节规划.md`、`plot/伏笔记录.md`、`plot/时间线.md`

### 1.4 结构化知识面板

值得吸收：

- 人物
- 世界观
- 时间线
- 剧情线
- 地点

原因：

- 这些分类和长篇连载记忆天然匹配

在本仓库的落点：

- 继续坚持 Markdown 项目记忆优先
- 后续如果做 UI，也只是这些文件的可视化编辑层，不另起孤岛存储

## 2. 只做可选层的能力

### 2.1 思维导图

可吸收，但只能做辅助层：

- 适合梳理多线关系、阵营站位、卷结构
- 不应替代文本总纲

落点原则：

- 可以辅助生成 `项目总纲` 或 `章节规划`
- 最终仍必须回写成线性文本和项目记忆文件

### 2.2 书名/简介生成

可吸收，但优先级不高：

- 它属于平台适配辅助，不是主线工作流核心
- 适合放在立项后、正文前的补充步骤

## 3. 明确不直接吸收什么

### 3.1 大而全题材提示词库

不建议直接吸收。

原因：

- 容易把写作退化成赛道套壳
- 数量多不等于质量高
- 会稀释本仓库现有的主线构建方法

更稳的做法：

- 保留少量高质量、可复用的操作模板
- 题材差异通过主线、冲突、回报和风格护栏体现，不靠堆 prompt 数量

### 3.2 AI 自评分自迭代

不建议作为主质检机制。

原因：

- 分数很容易奖励模板腔、解释腔和表面工整
- 不等于追读牵引、人物稳定和卖点兑现真的更强

更稳的做法：

- 评分只能做辅助参考
- 终审仍看一致性、钩子、补偿、关系推进和读者回报

### 3.3 硬编码模型路由和密钥

不吸收。

原因：

- 安全性差
- 可移植性差
- 和本仓库的 skill 资产没有关系

## 4. 当前仓库的吸收顺序

按优先级执行：

1. 先完善文本型流程资产
2. 再补脚本型辅助能力
3. 最后才考虑 UI 包装

当前优先项：

1. `references/quality/micro-revision-ops.md`
2. `references/planning/outline-refinement.md`
3. 未来的拆书提炼脚本
4. 未来的提示词工作台或可视化层

## 5. 判断一项外部能力值不值得吸收

至少问这 5 个问题：

- 它是提升结构稳定，还是只让界面更热闹
- 它是可复用资产，还是只适配某个项目
- 它能否落回我们的项目记忆文件
- 它会不会破坏“主线三步法 -> 逆推问题链 -> 章节执行”的主顺序
- 它是否真的减少返工，而不是制造新的维护成本

如果答不清，先不要吸收。
